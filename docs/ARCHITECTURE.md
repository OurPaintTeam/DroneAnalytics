# Техническое задание
## Общие принципы
 - изоляция
 - контроль взаимодействия
 - минимализация доверенной вычислительной базы
 - использование брокера сообщений, мониторов безропасности
 - Контейнеризация, IaC
## Базовые принципы (интерпретация в контексте настоящей компоненты)
 - Взаимодействие между доменами безопасности происходит только через брокер сообщений
 - Выдача событий безопасности по запросу.
 - Передача телеметрик в сервис аналитики
 - Передача событий безопасностей в сервис аналитики
 - События должны записываться в журнал.
 - Сертификаты будут выдаваться и подписываться Регулятором. (То есть не сами подписываем TLS, а получаем его - Вроде)
## Основные функции

### Базовые требования
- Система предполагает использование нескольких доменов безопасности, если особенности системы не предполагают иное (см. Брокер)
- Взаимодействие между доменами безопасности должно происходить только через брокер сообщений и проверяться монитором безопасности на соответствие политикам безопасности.
- Для системы должны быть перечислены и проанализированы активы, сформулированы цели и предположения безопасности, определены уровни доверия всех доменов безопасности.
- Для проверки политик безопасности должны быть реализованы автоматизированные тесты.
- Код доверенных доменов безопасности должен быть покрыт тестами не менее чем на 60%.
- Код доверенных доменов безопасности, повышающих целостность данных, должен быть покрыт тестами не менее чем на 70%, включая зависимости.
Все зависимости должны быть проанализированы на предмет известных уязвимостей. В доверенных доменах безопасности запрещается использовать сторонние пакеты с известными уязвимостями.
Оговорка: т.к. проект учебный, в отдельных случаях допускаются исключения для тестирования зависимостей, в таком случае должен быть проведён статический анализ кода, проанализирована информация в публичных банках данных угроз (например, БДУ ФСТЭК)
- Код остальных (недоверенных) доменов безопасности должен быть протестирован не менее чем на 30% (без учёта зависимостей).
- Выдача целей безопасности по запросу в формате сертификата системы, подписанного Регулятором (если применимо) 
- Выдача событий безопасности по запросу (если применимо)
- Передача телеметрии в сервис аналитики
- Передача событий безопасности в сервис аналитики
- Система (за исключением вспомогательных типа SITL) должна иметь сертификат безопасности с указанием целей безопасности (если применимо). Исключения допускаются для сторонних систем типа брокеров сообщений, баз данных и т.п. Сертификат безопасности должен быть подписан Регулятором / Авторизованным Сертифицирующим Центром (АСЦ)
- События должны записываться в журнал (с указанием уровня критичности)
### Инфопанель
- Отображение наличия узлов в сети (см. диаграмму системы систем)
- Отображение списка выполняемых заказов с их свойствами (задача, цели безопасности, какой эксплуатант и дрон выполняет)
- Отображение состояния счетов основных игроков (Агрегатор, Эксплуатант, Страховая компания, Разработчик, Регулятор, ОрВД БАС, Заказчик)
- Отображение количества выполненных заказов
- Oтображение количества инцидентов с нарушением целей безопасности
### Журнал событий (K11)
 - Инициализация хранилища для записи событий 
 - Запись нового события
 - Выгрузка записанных событий
### Аналитика (полученная потом и кровью)
 - метаданные по командам
 - по воздуху и в системе систем. Будет какой то серч, нужно будет отрабатывать какие соб. безоп. и т.д. Сколько каких систем, заказов и
потом эту инфу было бы не плохо визуализировать на инфопанели 
# Архитектура
## Фронтенд
Сайт на React или десктоп на PyQT. Цель - инфопанель. Визуализация аналитики. Текущее состояние систем. Визуализация
логов, их агрегация и фильтрация. Информирование о событиях безопасности.
## Бэкенд
На FastAPI. Обслуживает фронтенд и журнал.
## БД - ElasticSearch
Журнал событий.(логов)
## БД - PostgreSQL
На будущее, для хранения аналитики команд, их достижений и т.д.
# Контейнеры
