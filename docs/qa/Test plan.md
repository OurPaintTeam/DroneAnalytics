# Цель
Обеспечить системную проверку качества DroneAnalytics по направлениям:
- Корректность API и UI
- Соответствие требованиям безопасности
- Корректность интеграции сервисов.
# Область тестирования
## Backend
- API авторизации
- API логов
- Валидация данных
- Обработка ошибок
- API-key/JWT-проверки в защищённых маршрутах.
## Frontend
- доступ к страницам только после авторизации
- корректность экранов логов: загрузка данных, отображение состояний
- фильтры и пагинацию: изменение параметров должно приводить к корректному API-запросу
- базовую безопасность отображения пользовательских данных (без выполнения внедрённого скрипта)
- возможность XSS атаки
## Proxy
- маршрутизацию запросов с единой точки входа к backend;
## Elastic
- Backend делает записи в базу и может читать данные из неё
- Модификация журнала из вне
## E2E тесты
- Полная проверка действий пользователя.
## Вне области
- Кросс-браузерная проверка
- UX, UI тестирование
# Стратегии тестирования
## Приоритет
Практическое правило приоритета:
### P0
#### Backend
- API авторизации (логин, токены, refresh)
- Проверка доступа (JWT/API-key)
- Базовые операции с логами:
    - создать запись
    - получить запись (GET)
- Критичная валидация:
    - не пропускает явно некорректные данные
- Ошибки уровня:
    - 500 на базовых эндпоинтах
    - падение сервиса
#### Frontend
- доступ к приложению только после авторизации
- базовая загрузка логов:
    - экран открывается
    - данные приходят
#### Proxy
- backend доступен через proxy
#### Elastic
- backend может:
    - записывать лог
    - читать лог
#### E2E
- базовый пользовательский сценарий:
    1. логин
    2. Получение логов
### P1
#### Backend
- фильтрация логов
- пагинация
- корректные коды ошибок
- негативные сценарии API
- обработка граничных случаев
#### Frontend
- фильтры реально меняют API-запрос
- пагинация работает корректно
- корректное отображение данных
- XSS атаки
#### Elastic
- данные не меняются из вне злоумышленником
#### E2E
- фильтрация логов
## Стратегии по уровням
1. **Базовая работоспособность**. Что все сервисы поднялись, есть доступ к endpoints.
2. **API уровень**. Проверка работы API запросов backend.
3. **Сценарный уровень**. Проверка реальных сценариев работы.
4. **Уровень безопасности**. Проверка, что все 5 угроз не стали выполняться
5. **Инфраструктурный уровень**. Проверка работы proxy. 
## Регрессионная стратегия
После изменений кода надо проверить:
1. Либо **Короткая регрессия** - все из приоритета P0.
2. Либо **Полная проверка** - проверка P0 и P1
## Стратегия негативного тестирования
Для каждого положительного сценария, нужен и негативный сценарий.
Для API:
- Невалидный запрос
- Пропущены обязательные поля
- Невалидные токены
## Стратегия тестовых данных
В тестах будут использоваться:
1. Минимальные валидные данные: payload, api key.
2. Отдельно данные для негативных тестов.
Важно, чтобы эти данные не зависели от внешних источников и были изолированы друг от друга, чтобы не оказывать влияния на тесты.
## Стратегия управления окружением
Так как система контейнерная и зависит от запуска Elastic.
- учитывается задержка старта Elastiс
## Стратегия автоматизации
Базово: запуск тестов `make tests`.

CI-пайплайн:
1. Поднять окружение
2. Выполнить интеграционные тесты для endpoints
3. Выполнить security и proxy тесты
## Стратегия приемки дефектов
После каждого найденного дефекта:
1. Фиксируем наличие дефекта
2. добавляем/обновляем автотест. Чтобы учитывать этот дефект
3. исправляем код
4. подтверждаем, что новый тест стал зелёным и регрессия не сломалась
# Типы тестов
Перечислим типы тестов, которые должны применяться в проекте
## Тесты работоспособности
**Цель:** быстро понять, что система в принципе работоспособна после старта/изменений.
**Проверяют**:
- контейнеры поднялись и ключевые сервисы отвечают;
**Когда запускать:**
- первым шагом в CI
- перед запуском полного регресса.
## Функциональные API-тесты (endpoints)
**Цель:** проверить, что endpoints API работают.
**Что проверяют:**
- корректность сценариев авторизации (auth)
- Корректность сценариев получения и записи логов (log)
- фильтрации, пагинацию.
- корректность формата ответа и HTTP-кодов.
**Типичные кейсы:**
 - позитивные
 - негативные
## Интеграционные тесты
**Цель:** проверить взаимодействие компонентов между собой, а не только отдельный endpoint.
**Что проверяют:**
- связку `backend ↔ elastic` (запись/чтение журналов);
- связку `proxy ↔ backend`;
- корректность работы auth в реальной цепочке запросов.
**Признак ценности:**
- выявляют проблемы конфигурации, сетевого взаимодействия и несовместимости контрактов.
## Security-тесты
**Цель:** проверить отсутствие регрессий по критичным угрозам.
**Что проверяют:**
- spoofing API key
- риски компрометации JWT-секрета
- целостность журналов
- XSS-сценарии в отображении данных
- базовую устойчивость к DoS-подобным воздействиям
**Когда обязательны:**
- перед релизом;
- после изменений в auth/security/валидации и обработке пользовательского ввода.
## Негативные тесты
**Цель:** подтвердить корректное поведение системы при неверных входных данных и нештатных запросах.
**Что проверяют:**
- невалидные тела запросов
- отсутствующие обязательные поля
- неверные типы/форматы
- неавторизованный доступ
- обращение к несуществующим маршрутам.
**Ожидаемый результат:**
- контролируемая ошибка
- единая структура error response
## Тесты выгрузки данных
**Цель:** проверить корректность экспорта логов для внешнего использования.
**Что проверяют:**
- доступность download endpoint
- корректность формата файла/контента
- корректную фильтрацию экспортируемых данных
- обработку пустой выборки
## Proxy-тесты
**Цель:** проверить, что единая точка входа маршрутизирует запросы правильно
**Что проверяют:**
- проксирование в backend по ожидаемым путям
## Frontend тесты
**Цель:** проверить корректность работы frontend
**Что проверяют:**
- frontend запускается
- frontend корректно отправляет запросы на backend
# Ресурсы тестирования
## Команда и роли
На текущем этапе в проекте участвует **1 тестировщик**.
Распределение ответственности:
- **Тестировщик (1 человек):**
1. планирование и актуализация тест-плана
2. подготовка и запуск тестовых прогонов
3. оформление баг-репортов
4. контроль качества релиз-кандидата по критериям выхода
5. добавление/обновление автотестов на критичные баги
- **Разработчики (поддерживающая роль):**
1. исправление дефектов
## Технические ресурсы
Минимально необходимые ресурсы для запуска тестов:
- Docker + Docker Compose;
- доступ к репозиторию и конфигам (`configs/*`, `secrets/*`);
- локальная машина/CI-раннер с достаточными CPU/RAM для контейнеров backend/frontend/elastic/proxy.
# Тестовое окружение
## Ручное тестирование
1. Создание секретов
2. Запуск всех сервисов
Реализуется через `make local`
## CI-окружение
1. Создание секретов
2. Запуск всех сервисов
3. Запуск сервиса с тестами
Реализуется через `make tests`
# Критерии начала и окончания тестирования для конкретной версии проекта.
## Начало
Чтобы начать тестирования должны быть готовы:
1. Написаны требования. Есть примеры того, как все должно работать.
2. Готова полная версия ПО. Об этом сообщено тестировщику.
## Окончание
Тестирование для версии ПО считается выполненным если:
1. все запланированные тесты
2. код соответствует требованиям
3. нет критических багов (P0)
4. Допустимое количество багов для P1
По выявленным дефектам следует написать отчет.
# Риски и допущения
## Риски
1. Ограниченность ресурсов - один тестировщик.
2. Изменение требований по окончании тестирования. Решение: уточнять требования по мере тестирования в дополнении к фиксации требований
## Допущения
1. Proxy - единственная точка входа
2. Backend соответствует API-контракту
3. Elastic, proxy и другие сервисы работают стабильно
4. Тесты не влияют друг на друга
5. Пользователь действует по основным сценариям
6. Способы шифрования используемые в проекте - безопасны
7. Пользователь использует поддерживаемый браузер
