# 1. Проверка API ключа
Проверка показала, что если злоумышленник сможет получить API ключ, то он спокойно может отправлять с его помощью логи в базу через proxy.
# 2. Проверка Security ключа
Проверка показала, что если злоумышленник сможет получить Security ключ, то он спокойно может использовать его для создания токенов авторизации. К тому же можно делать запросы от пользователя и без авторизации.
# 3. Взаимодействие с Elastic
Проверка показала, что при наличии доступа к базе ElasticSearch злоумышленник может делать на неё запросы, которые могут менять содержимое базы.
# 4. Проверка попытки Dos атаки
Проверка показала, что в проекте нет ограничение на количество запросов в базу. Но при умеренном числе запросов система не падает и дает залогиниться.
# 5 Проверка XSS атаки
Проверка показала, что попытка отправить script или javascript URI данные остаются данными и в базе ElasticSearch отражаются как данные. Есть базовые меры защиты для HTTP.
Проверка в браузере mozilla firefox показала, что уязвимость XSS не подтвердилась.
Были попытки отправить разные вредоносные строки через POST /api/log/*, но на сайте они отображались только как обычный текст.
Сценарии с \<script\>, onerror, javascript:, а также с SVG и “замаскированными” вариантами не выполнили JavaScript-код в интерфейсе.
Это значит, что в проверенных случаях фронтенд не исполняет содержимое поля message как код.
